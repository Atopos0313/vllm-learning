# vLLM Nightly + Qwen2.5-0.5B 本地部署实践

> 2026-07-06 完成于WSL2 环境个vLLM nightly 的独立部署、`Qwen/Qwen2.5-0.5B-Instruct` 本地模型加载、OpenAI-compatible API Server 启动验证，并沉混于RTX 2080 Ti 地FA2、BF16、FP8 等能力上的兼容性处理方式　
## 1. 学习目标

- 搭建可复现的 vLLM nightly 实验环境　- 验证 WSL2 个PyTorch、CUDA、RTX 2080 Ti 的可用性　- 就Hugging Face 模型下载到本地，避免服务启动阶段被网络阻塞　- 使用 vLLM 启动 OpenAI-compatible API Server　- 记录旧架果GPU 地vLLM 高性能后端中的限制与处理方案　
## 2. 环境与模型
| 项目 | 配置 / 结果 | 说明 |
|------|-------------|------|
| 系统环境 | Ubuntu 26.04 / WSL2 | Windows 下的 Linux 推理环境 |
| Python | `3.12.3` | 使用独立虚拟环境 |
| 环境管理 | `uv venv` | 虚拟环境位于当前项目目录 |
| vLLM | `0.24.0 nightly` | 通过 nightly wheel 安装 |
| GPU | `NVIDIA GeForce RTX 2080 Ti` | Turing 架构，compute capability 7.5 |

| 模型 | `Qwen/Qwen2.5-0.5B-Instruct` | 本地路径加载 |

## 3. 实现过程

### 3.1 创建工作目录与虚拟环境
```bash
cd ~/llm/vllm-nightly
uv venv --python 3.12 .venv
source .venv/bin/activate
```

> `uv` 创建的虚拟环境在当前文件夹内，后续启动前需要先 `cd` 到对应目录；如果删除源文件夹，虚拟环境也会消失　
### 3.2 安装 vLLM nightly

```bash
uv pip install -U vllm \
  --torch-backend=auto \
  --extra-index-url https://wheels.vllm.ai/nightly

uv pip install -U huggingface_hub pandas
```

### 3.3 验证 vLLM / CUDA / GPU

```bash
python -c "import vllm; print('vllm', vllm.__version__, vllm.__file__)"
python -c "import torch; print('cuda', torch.cuda.is_available(), torch.cuda.get_device_name(0))"
```

验证结果
```text
vllm 0.24.0 /home/atopos/llm/vllm-nightly/.venv/lib/python3.12/site-packages/vllm/__init__.py
cuda True NVIDIA GeForce RTX 2080 Ti
```

结论：vLLM 安装成功，PyTorch 可调用CUDA，WSL2 内部可识到RTX 2080 Ti　
## 4. 模型下载策略

直接使用远程模型名启动时
```bash
vllm serve Qwen/Qwen2.5-0.5B-Instruct
```

服务长时间卡在模型加载阶段。结合日志判断，`config`、`tokenizer` 等小文件可以较快下载，但权重文件下载较慢，因此改为提前下载模型到本地目录　
```bash
mkdir -p ~/llm/models/Qwen2.5-0.5B-Instruct

hf download Qwen/Qwen2.5-0.5B-Instruct \
  --local-dir ~/llm/models/Qwen2.5-0.5B-Instruct
```

如Hugging Face 下载较慢，可切换镜像源：

```bash
export HF_ENDPOINT=https://hf-mirror.com

hf download Qwen/Qwen2.5-0.5B-Instruct \
  --local-dir ~/llm/models/Qwen2.5-0.5B-Instruct
```

模型目录重点检查：

| 文件 | 作用 |
|------|------|
| `config.json` | 模型结构配置 |
| `generation_config.json` | 生成参数配置 |
| `model.safetensors` | 模型权重文件 |
| `tokenizer.json` | tokenizer 配置 |
| `tokenizer_config.json` | tokenizer 元信息|
| `vocab.json` / `merges.txt` | 分词词表文件 |

## 5. vLLM 服务启动

```bash
vllm serve ~/llm/models/Qwen2.5-0.5B-Instruct \
  --served-model-name qwen-0.5b \
  --host 0.0.0.0 \
  --port 8000 \
  --dtype float16 \
  --max-model-len 1024 \
  --gpu-memory-utilization 0.60
```

| 参数 | 当前候| 选择原因 |
|------|--------|----------|
| 模型路径 | `~/llm/models/Qwen2.5-0.5B-Instruct` | 使用本地模型，避免启动时访问 Hugging Face |
| `--served-model-name` | `qwen-0.5b` | 对外暴露算洁模型名 |
| `--host` | `0.0.0.0` | 允许 WSL2、宿主机或局域网访问 |
| `--port` | `8000` | 使用默认测试端口 |
| `--dtype` | `float16` | RTX 2080 Ti 不支持BF16 |
| `--max-model-len` | `1024` | 降低 KV cache 显存占用 |
| `--gpu-memory-utilization` | `0.60` | 保守使用显存，优先保证服务稳定|

## 7. 问题与沉混

| 现象 | 原因分析 | 处理方式 | 状性|
|------|----------|----------|------|
| 启动提示 FA2 不支持| RTX 2080 Ti compute capability 个7.5，低于FA2 要求 | 接受 vLLM fallback 到`TRITON_ATTN` | 已处理|
| BF16 / FP8 不可用| 2080 Ti 不支持BF16 / FP8 推理路径 | 显式使用 `--dtype float16` | 已处理|
| FlashInfer sampler warning | 部分 sampler 需要更新架果GPU 支持 | 可选设置`VLLM_USE_FLASHINFER_SAMPLER=0` | 可选优化|
| 服务卡在模型加载 | Hugging Face 权重下载慢| 使用 `hf download` 预下载模型，从本地路径启动| 已解决|

## 8. 可复用经验
1. vLLM 部署前先验证 `torch.cuda.is_available()` 和GPU 名称，避免把环境问题误判为服务问题　2. 模型下载与服务启动应分开排查，本地模型路径更适合复现实验　3. 对RTX 2080 Ti 这类旧架果GPU，应优先使用 `float16`，并接受部分高性能后端 fallback　4. 首次验证可使用较小`max-model-len` 和较低`gpu-memory-utilization`，跑通后再逐步调大
