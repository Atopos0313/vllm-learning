# DSpark Speculative Decoding 部署验证与原理沉混

> 2026-07-07 完成于`dspark-exp` 地WSL2 + RTX 2080 Ti 环境下的 Baseline 个DSpark 两条推理链路验证，确认speculative decoding 指标实际生效，并系统整理于KV cache、TTFT、TPOT、throughput 和投机解码适用条件　
## 1. 学习目标

- 在本地环境跑通`ikaroso/dspark-exp`　- 准备 `Qwen3-4B` target model 个`dspark_qwen3_4b_block7` draft model　- 分别启动 Baseline serve 和DSpark serve　- 通过进程、日志、接口和 `/metrics` 确认 speculative decoding 是否实际启用　- 建立 Phase1 验收流程，包拟smoke、TTFT probe、metrics 检查和 `record_sweep.sh` 结果记录　- 梳理大模型推理性能指标和投机解码原理

## 7. DSpark 生效证据

仅看 `/v1/models` 不能判断 DSpark 是否启用，需要结合进程参数、日志和 metrics　
```bash
ps -ef | grep -E "vllm serve|speculative-config|dspark_qwen3_4b_block7" | grep -v grep

curl -s http://127.0.0.1:8001/metrics | grep -Ei "spec|dspark|draft|accept"
```

已观察到的指标：

```text
vllm:spec_decode_num_drafts_total 15.0
vllm:spec_decode_num_draft_tokens_total 105.0
vllm:spec_decode_num_accepted_tokens_total 17.0
vllm:spec_decode_num_accepted_tokens_per_pos_total{position="0"} 11.0
vllm:spec_decode_num_accepted_tokens_per_pos_total{position="1"} 3.0
vllm:spec_decode_num_accepted_tokens_per_pos_total{position="2"} 2.0
vllm:spec_decode_num_accepted_tokens_per_pos_total{position="3"} 1.0
```

日志摘要＀
```text
SpecDecoding metrics:
Mean acceptance length: 2.13
Accepted: 17 tokens
Drafted: 105 tokens
Per-position acceptance rate: 0.733, 0.200, 0.133, 0.067, 0.000, 0.000, 0.000
Avg Draft acceptance rate: 16.2%
```

结论＀
1. DSpark speculative decoding 已实际工作　2. 当前短测接受率约 `16.2%`，说明draft model 有一定命中，但接受率仍偏低　3. `num_speculative_tokens=7` 可以跑通，但后几个位置接受率为 0，后续应通过 benchmark 判断是否需要降低该参数　4. smoke 测试不能直接证明 DSpark 更快，后续需要继续记录TTFT、TPOT、throughput、显存占用等指标　
## 8. 推理原理沉混

| 概念 | 说明 | 与本次实验的关系 |
|------|------|------------------|
| Prefill | 处理输入 prompt，建立上下文表示和KV cache | 主要影响 TTFT |
| Decode | 通token 生成输出 | 投机解码主要优化该阶段|
| KV cache | 缓存历史 token 的key/value 中间结果 | 提升速度但占用显存|
| TTFT | Time To First Token | 不一定能体现 DSpark 收益 |
| TPOT | Time Per Output Token | 更适合观察 decode 优化 |
| Throughput | tokens/s | 需要结启batch、并发、硬件利用率分析 |
| Speculative decoding | draft model 先猜多个 token，target model 再验证| 接受率越高，越可能减少target model 串行 decode 次数 |

投机解码更容易加速的场景＀
1. 输出较长　2. `temperature` 较低　3. 文本格式稳定　4. draft model 个target model 行为接近　5. GPU 还没有被高并发batch 完全填满　
不一定有收益的场景：

1. 输出很短　2. `temperature` 较高　3. 开放式创作　4. draft model 接受率低　5. draft model 本身带来的额外计算抵消收益　
## 9. Phase1 验收流程

两个终端分工＀
| 终端 | 职责 | 内容 |
|------|------|------|
| 终端 A | 启动服务 | 启动 Baseline 成DSpark serve，占用GPU |
| 终端 B | 请求与记录| 查询模型、跑 TTFT probe、检查metrics、写入实验记录|

公共环境＀
```bash
cd ~/dspark-exp
source .venv/bin/activate
export VLLM_WSL2_ENABLE_PIN_MEMORY=1
export HF_HUB_OFFLINE=1
export HF_HUB_DISABLE_XET=1
unset HF_HUB_ENABLE_HF_TRANSFER
```

记录 baseline＀
```bash
scripts/record_sweep.sh \
  --id phase1_smoke_baseline \
  --layer L0 \
  --knob smoke \
  --value ok \
  --mode baseline \
  --ttft-ms <PROBE_TTFT_MS> \
  --risk none \
  --notes "Phase1工具验收 baseline local" \
  --config configs/reference_baseline.local.yaml
```

记录 DSpark＀
```bash
scripts/record_sweep.sh \
  --id phase1_smoke_dspark_local \
  --layer L0 \
  --knob smoke \
  --value ok \
  --mode dspark \
  --ttft-ms <PROBE_TTFT_MS> \
  --risk none \
  --notes "Phase1工具验收 dspark local" \
```

## 10. 问题与沉混

| 问题 / 现象 | 原因分析 | 处理方式 | 状性|
|-------------|----------|----------|------|
| PyPI 正式版vLLM 不含 DSpark | DSpark 能力依赖 nightly 构建 | 使用 nightly wheel 索引安装 | 已处理|
| Hugging Face Xet / CAS 401 | 在线访问和Xet/CAS 权限可能影响模型下载或加载| 设置 `HF_HUB_DISABLE_XET=1`，本地实验使用`HF_HUB_OFFLINE=1` | 已处理|
| `/v1/models` 返回本地路径 | vLLM 使用本地模型路径启动时，served model id 默认是路径| API 请求中使用`/home/atopos/llm/models/Qwen3-4B` | 已处理|
| 使用 `Qwen/Qwen3-4B` 请求可能 404 | 请求 model 与服务注况id 不一致| 使用 `/v1/models` 返回的完整id，或显式设置 `--served-model-name` | 已处理|
| Baseline 个DSpark 不能同时运行 | 单卡 GPU 显存和端口资源有限| 切换前清理旧服务并检查`nvidia-smi` | 已处理|
| 单看 `/v1/models` 无法判断 DSpark 是否启用 | 该接口只展示 target model | 结合进程参数、日志与 `/metrics` 判断 | 已明确|
| DSpark 接受率仅约`16.2%` | 当前 prompt 个draft 策略下接受比例较低| 后续通过 benchmark 和参数对比判文| 待验证|
| TTFT 不一定体环DSpark 收益 | TTFT 主要可prefill 影响 | 后续重点观察 TPOT、tokens/s 和inter-token latency | 待验证|
